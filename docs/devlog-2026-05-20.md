## 1. Send Feedback (branch: user-roles → merged to main)

### Goal
Allow users to send feedback from both the Productions page and the PlayBlocker
workspace without seeing To/From/Subject fields.

### What was built

**Backend — `NodeMySql/app.js`**
- Added `resend` npm package and `const { Resend } = require('resend')`
- `RESEND_API_KEY` added to `.env` (gitignored; must be set manually on server)
- `POST /api/feedback` endpoint: requires auth (`verifyToken`), accepts `{ message }`,
  sends via Resend with:
  - From: `PlayBlocker Feedback <noreply@lwebber.ca>` (Resend requires verified sender domain)
  - Reply-To: logged-in user's email (`req.user.email` from JWT)
  - To: `leighwebber@live.com`
  - Subject: `PlayBlocker feedback`

**Frontend — `Modules/Feedback.js`** (new shared module)
- Exports `openFeedbackModal()` — dynamically builds and appends a modal overlay
- Modal contains: heading, textarea, inline status message, Cancel and Send buttons
- Send POSTs to `/api/feedback`, shows "Feedback sent. Thank you!" then auto-closes
- Clicking outside the modal or pressing Escape closes it
- Imported by both `Productions.js` and `PlayBlocker.js`

**UI**
- "Send Feedback" button added to Productions page header (`.link-btn secondary`)
- "Send Feedback" button added to PlayBlocker header, positioned between Save and ?
- CSS: `#feedback-btn` positioned absolutely in PlayBlocker header (`right: 215px`)

**Bug fix — keyboard trap**
- PlayBlocker's `handleKeyDown` was swallowing Space, Tab, Arrow keys etc. inside
  the feedback textarea
- Fix: added a two-line guard at the top of `handleKeyDown`:
  ```javascript
  const tag = event.target?.tagName;
  if (tag === 'TEXTAREA' || tag === 'INPUT' || event.target?.isContentEditable) return;
  ```
- This guard applies to all current and future text inputs on the PlayBlocker page

**Help page**
- Added "Send Feedback" section to `Pages/help.html` under a new "Other" nav group
- Three-step workflow, note that user's email is included automatically for replies,
  tip that Escape closes the modal

## 2. Roles Feature (branch: roles — in progress)

### Architecture decisions made

| Decision | Choice | Rationale |
|---|---|---|
| Role model | Hierarchical (one role per user per production) | Simpler than ACL; covers all current needs |
| AppOwner storage | `is_app_owner` flag on `user` table | Not production-scoped, so wrong table for `production_user_role` |
| Future multiple roles | Deferred | Schema can migrate later if Influencer-type role arrives |
| 2FA | Deferred to Phase 2 | Build registration/roles first |
| Audit log | Deferred | Separate `audit_log` table later; don't pollute existing tables |
| DB version control | Manual `migrations/` SQL files committed to git | Lightweight, sufficient at current scale |
| SMS 2FA provider | Twilio (when needed) | ~$0.0079/msg, industry standard |

### Role hierarchy

```
AppOwner (superuser flag on user, not in production_user_role)
  └── Owner   (role_id = 1) — full access, can grant Editor/Viewer
        └── Editor  (role_id = 2) — can edit, cannot delete or manage roles
              └── Viewer  (role_id = 3) — read-only
```

### Database migrations (run on startup via `app.js` db.connect callback)

```sql
ALTER TABLE user ADD COLUMN email_verified TINYINT(1) NOT NULL DEFAULT 1;
ALTER TABLE user ADD COLUMN verification_token VARCHAR(255);
ALTER TABLE user ADD COLUMN verification_token_expires DATETIME;
ALTER TABLE user ADD COLUMN is_app_owner TINYINT(1) NOT NULL DEFAULT 0;
ALTER TABLE production_user_role ADD PRIMARY KEY (user_id, production_id);
INSERT IGNORE INTO role (id, role_name) VALUES (1,'Owner'),(2,'Editor'),(3,'Viewer');
INSERT IGNORE INTO production_user_role (user_id, role_id, production_id)
    SELECT user_id, 1, id FROM production;   -- backfill existing productions
```

Note: `email_verified DEFAULT 1` so existing users are not locked out by the migration.

### New API endpoints

| Method | Path | Auth | Description |
|---|---|---|---|
| POST | `/api/register` | none | Creates unverified user, sends confirmation email |
| GET | `/api/verify-email?token=` | none (token = credential) | Verifies email, redirects to `/index.html?verified=true` |
| POST | `/api/resend-verification` | none | Issues fresh token; always returns neutral message |
| GET | `/api/productions` | JWT | Role-filtered; AppOwner sees all with `owner_email` |
| POST | `/api/productions` | JWT | Creates production + auto-assigns Owner role |
| GET | `/api/productions/:id/roles` | JWT (Owner/AppOwner) | Lists users+roles for a production |
| POST | `/api/productions/:id/roles` | JWT (Owner/AppOwner) | Grants Editor or Viewer role by email |
| DELETE | `/api/productions/:id/roles/:userId` | JWT (Owner/AppOwner) | Revokes role (cannot revoke Owner) |

## 3. Speaker Autofill (branch: roles)

### Goal
Let users immediately open a freshly-uploaded script in PlayBlocker without
manually filling in speaker details first.

### Autofill logic (`autofillSpeakers()` in `Productions.js`)
- Only fills blank fields — existing values are never overwritten
- **Names**: cycles through 25 whimsical pre-defined first/last name pairs
  (Horatio Pumpernickel, Evangeline Cholmondeley, etc.)
- **Initials**: takes the script name's first letter, then pairs it with each
  subsequent letter until a unique combination is found; falls back to
  `first-letter + digit` if all letter pairs are exhausted
- **Colors**: cycles through 20 curated dark CSS named colors
  (`darkred`, `darkblue`, `teal`, `indigo`, `saddlebrown`, …) — dark colors
  chosen because shadow divs display as a paler variant

### Save validation
- Completely blank rows (all four fields empty) are silently skipped
- Blocks save if any included speaker is missing initials (shows which ones)
- Blocks save if initials are not unique across speakers
- Status messages are inline (green/red paragraph below buttons) — no more `alert()`
